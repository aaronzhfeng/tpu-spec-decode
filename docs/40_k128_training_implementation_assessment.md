# Doc 40: K=128 Training Implementation Assessment

## Purpose

This document assesses the feasibility and concrete requirements for training a DFlash drafter with block_size=128 **without access to the original z-lab/dflash training code**. It inventories what exists in this repo, what must be built, identifies the tricky parts, and provides a realistic timeline.

---

## 1. What DFlash Training Actually Does

DFlash training is conceptually simple: **masked token prediction conditioned on target model hidden states**. The training loop works as follows:

1. Take a (prompt, response) pair where the response was generated by the target model (Qwen3-4B)
2. Run the full clean sequence through the **frozen** target model to extract hidden states from 5 uniformly sampled layers
3. Randomly sample anchor positions in the response; mask the next K-1 tokens after each anchor
4. Concatenate all blocks into one sequence with a sparse attention mask (bidirectional within blocks, no cross-block attention)
5. Run the draft model forward: embed masked tokens via the target's embedding layer, inject projected target hidden states into every draft layer's KV, predict logits via the target's LM head
6. Compute cross-entropy loss on masked positions with position-dependent exponential decay weights
7. Backprop through the draft model only (target model, embedding, and LM head are all frozen)

The training objective is standard masked LM — no exotic losses, no adversarial components, no reinforcement learning.

## 2. Architecture: What Gets Trained

### Trainable Parameters (Draft Model Only)

```
DFlashDraftModel
├── fc: Linear(num_target_layers × hidden_size, hidden_size)   ← projects target features
├── hidden_norm: RMSNorm(hidden_size)                          ← normalizes projected features
├── 5 × DFlashDecoderLayer:
│   ├── input_layernorm: RMSNorm
│   ├── self_attn:
│   │   ├── q_proj: Linear(hidden_size, num_heads × head_dim)
│   │   ├── k_proj: Linear(hidden_size, num_kv_heads × head_dim)
│   │   ├── v_proj: Linear(hidden_size, num_kv_heads × head_dim)
│   │   ├── o_proj: Linear(num_heads × head_dim, hidden_size)
│   │   ├── q_norm: RMSNorm(head_dim)
│   │   └── k_norm: RMSNorm(head_dim)
│   ├── post_attention_layernorm: RMSNorm
│   └── mlp (SwiGLU):
│       ├── gate_proj: Linear(hidden_size, intermediate_size)
│       ├── up_proj: Linear(hidden_size, intermediate_size)
│       └── down_proj: Linear(intermediate_size, hidden_size)
└── norm: RMSNorm(hidden_size)                                 ← final norm
```

### Frozen (Shared from Target)

- `embed_tokens`: Token embedding layer from Qwen3-4B
- `lm_head`: Language model head from Qwen3-4B
- `rotary_emb`: RoPE embeddings (recomputed, not stored)

### Estimated Trainable Parameter Count

For Qwen3-4B (hidden_size=2560, intermediate_size=6912, num_heads=32, num_kv_heads=8, head_dim=128):

| Component | Params per layer | Total |
|-----------|-----------------|-------|
| Q/K/V/O projections | ~26.2M | ~131M (5 layers) |
| MLP (gate+up+down) | ~53.1M | ~265M (5 layers) |
| Norms | ~15K | ~75K (5 layers) |
| FC projection | 5 × 2560 × 2560 = ~32.8M | ~32.8M |
| **Total trainable** | | **~429M** |

This is small enough to train on a single TPU v4-8 (4 chips) without model parallelism.

## 4. The Tricky Parts

### 4.1 Block Construction and Sparse Attention Mask

**What it is:** During training, multiple blocks are packed into a single sequence. Each block has an anchor token (clean) followed by K-1 masked positions. Blocks must NOT attend to each other — only bidirectionally within themselves and to their target context.

**Why it's tricky:** The original code uses PyTorch Flex Attention (`torch.nn.attention.flex_attention`) to implement the sparse block-diagonal attention pattern efficiently. This is a relatively new PyTorch feature (Dec 2024). On TPU via PyTorch/XLA, Flex Attention may not be supported.

**Alternatives:**
- **Dense attention + explicit mask tensor:** Construct a full (seq_len × seq_len) boolean mask. Works everywhere but wasteful for long sequences with many blocks.
- **Loop over blocks individually:** Process each block as a separate batch element. Simpler but slower and changes the gradient dynamics (no cross-block gradient averaging within a sequence).
- **JAX vmap + custom mask:** If training in JAX, use `jax.vmap` over blocks with a fixed-size mask.

**Recommendation:** Start with dense attention + explicit mask. For K=128 with ~60 anchors per 3072-token sequence, the sequence length after concatenation is ~60 × 128 = 7,680 tokens. The mask tensor is 7,680² × 1 bit ≈ 7 MB per sample — manageable.

### 4.2 KV Injection Mechanics

**What it is:** Target hidden states are not just an input to the first layer. They're projected through `fc` + `hidden_norm`, then used as Key/Value context in **every** draft layer. During training, the target hidden states for all tokens in the sequence are available, so each block's KV injection uses the target features corresponding to that block's position range.

**Why it's tricky:** In inference, KV injection happens via the KV cache mechanism. In training, there is no KV cache — the model processes all blocks in parallel. The target hidden states must be correctly sliced per-block and broadcast into each layer's attention as the K/V context component.

**The existing PyTorch code shows the pattern clearly** (`zhongyan_dev/dflash/model/dflash.py:73-78`):

```python
k_ctx = self.k_proj(target_hidden)   # context K from target features
k_noise = self.k_proj(hidden_states)  # noise K from draft embeddings
k = torch.cat([k_ctx, k_noise], dim=1)  # concatenate along seq dim
```

During training, `target_hidden` per block = projected target features at the anchor's position (replicated across the block). This is the same `fc(concat(layer_hidden_states))` → `hidden_norm()` pipeline, just applied at training-time scale.

**Recommendation:** The forward pass code already handles this correctly for inference. For training, ensure the target hidden features are sliced per-block before being passed to each layer.

### 4.3 Loss Decay Tuning for K=128

**What it is:** Position-dependent loss weighting:

```
w_k = exp(-(k-1) / γ)
```

Published values: γ=7 for K=16, γ=5 for K=10, γ=4 for K=8.

**Why it matters for K=128:** With γ=7 and K=128, the weight at position 128 would be `exp(-127/7) ≈ 1.3e-8` — effectively zero. The model would barely train on positions beyond ~30. This might be fine (early positions matter most for acceptance), or it might undertrain the tail.

**The pattern from published values:**

| K | γ | γ/K ratio | Weight at last position |
|---|---|-----------|------------------------|
| 8 | 4 | 0.50 | exp(-7/4) = 0.17 |
| 10 | 5 | 0.50 | exp(-9/5) = 0.17 |
| 16 | 7 | 0.44 | exp(-15/7) = 0.12 |

The γ/K ratio is ~0.44-0.50, giving ~12-17% weight at the last position.

**Extrapolation for K=128:** To maintain similar tail weight, γ ≈ 0.47 × 128 ≈ 60. But this might over-flatten the weighting. The paper only tested up to K=16; K=128 is 8× larger and may need a different decay schedule entirely (e.g., linear decay, piecewise, or a plateau + decay).

**Recommendation:** Grid search γ ∈ {20, 40, 60, 80} for K=128. Also test uniform weighting as a baseline. Validate by measuring tau on a held-out set every 0.5 epochs.

### 4.4 Anchor Count and Training Signal Density

**What it is:** For K=16, the paper uses 512 anchors per 3072-token sequence, yielding ~512 × 15 = 7,680 masked positions per sequence.

**For K=128:** To maintain comparable total masked positions: 7,680 / 127 ≈ 60 anchors per sequence. But 3072 / 128 = 24 non-overlapping blocks max. With 60 anchors, blocks will heavily overlap.

**Options:**
1. **60 anchors, allow overlap:** Some positions masked by multiple blocks → duplicated loss signal. May cause over-fitting on densely-sampled regions.
2. **24 anchors, no overlap:** Clean block boundaries but fewer masked positions (24 × 127 = 3,048) — 60% less training signal per sequence.
3. **24 anchors + longer sequences:** Increase max_seq_len to 8192 to fit more non-overlapping blocks.

**Recommendation:** Start with 24 non-overlapping anchors (option 2). Compensate for reduced per-sequence signal by increasing epochs from 6 to ~10-12, or increasing dataset size. The clean block boundaries simplify the attention mask and avoid double-counting loss.

## 5. Implementation Paths

### Path A: PyTorch on GPU (Fastest to Working Code)

**Approach:** Use the existing PyTorch DFlash model (`zhongyan_dev/dflash/model/dflash.py`) directly. Write a standard PyTorch training loop with HuggingFace datasets + Trainer or a manual loop.

**Pros:**
- Model code already works in PyTorch
- FlashAttention2 available for non-causal attention
- Flex Attention available for sparse block masks
- Standard PyTorch debugging and profiling tools
- Can validate against published K=16 tau before scaling to K=128

**Cons:**
- Requires GPU access (A100/H100/H200)
- No direct TPU utilization during training
- Need to convert trained weights to JAX for TPU inference

**Estimated engineering time:** 1-1.5 weeks

### Path B: PyTorch/XLA on TPU

**Approach:** Same PyTorch model, but run via PyTorch/XLA on TPU. Use `torch_xla` for device placement and compilation.

**Pros:**
- Uses available TPU hardware directly
- Model code reusable from Path A
- Weight format compatible with JAX inference (both go through HF checkpoints)

**Cons:**
- Flex Attention likely unsupported on XLA → need manual mask implementation
- PyTorch/XLA has compilation quirks (dynamic shapes, recompilation triggers)
- Slower iteration during development (compilation overhead)

**Estimated engineering time:** 2-2.5 weeks

### Path C: Pure JAX/Flax Training on TPU

**Approach:** Port the training forward pass to JAX using the existing JAX inference model (`tpu-inference/tpu_inference/models/jax/dflash.py`) as a starting point. Write the training loop in JAX with Optax.

**Pros:**
- Native TPU execution, best performance
- Direct weight compatibility with inference code
- Full control over compilation and sharding

**Cons:**
- Largest engineering effort — inference model must be adapted for training (add gradient support, handle variable block counts, implement loss)
- JAX training patterns differ from PyTorch (functional transforms, `jax.grad`, etc.)
- The JAX model uses NNX, which has different training patterns than classical Flax

**Estimated engineering time:** 3-4 weeks

### Recommendation

**Path A (PyTorch on GPU) is the pragmatic choice.** The goal is to measure whether K=128 improves tau — this is a research question, not a systems engineering task. Getting to a trained checkpoint fastest is what matters. Once a K=128 checkpoint exists, the existing JAX inference pipeline can evaluate it on TPU immediately.

If GPU access is not available, **Path B** is the next best option.

## 6. Training Data Pipeline

### Data Requirements

| Item | Specification | Source |
|------|--------------|--------|
| Dataset | ~800K (prompt, response) pairs | NVIDIA Nemotron-Post-Training-Dataset-v2 + CodeAlpaca |
| Response source | Target-model-generated (not original) | Run Qwen3-4B on all prompts |
| Max sequence length | 3072 tokens | Paper Section A.1 |
| Target hidden states | 5 layers, uniformly sampled between layer 2 and layer (N-2) | For Qwen3-4B (36 layers): layers [2, 10, 18, 26, 34] (estimated) |

### Offline Caching (Recommended)

Precompute target hidden states for all training samples to avoid running the frozen target model during every training step.

**Storage estimate per sample:**
```
5 layers × 3072 tokens × 2560 hidden_size × 2 bytes (bf16) = ~75 MB per sample
800K samples × 75 MB = ~60 TB
```

This is prohibitively large for full caching. **Solutions:**
1. **Online training:** Compute target hidden states on-the-fly each step. Costs ~2× the forward pass compute but zero storage.
2. **Partial caching:** Cache only the projected features (after FC layer): 3072 × 2560 × 2 bytes = ~15 MB per sample → ~12 TB total. Still large.
3. **Shard caching:** Cache features for one epoch at a time, regenerate per-epoch (different anchor positions anyway).

**Recommendation:** Online training is simplest. The target model forward pass (Qwen3-4B) takes ~10ms on TPU v4; the draft model backward pass is the bottleneck anyway. The 2× overhead is acceptable for simplicity.

## 7. Validation Strategy

### Critical: Train K=16 First

Before investing 3-5 days training K=128, **reproduce the published K=16 results** with your training code. This validates:

- Block construction is correct
- KV injection works properly
- Loss weighting matches paper behavior
- Acceptance rates are in the right ballpark

**Success criterion:** Trained K=16 achieves tau ≥ 6.0 on Math500 (published: 6.33 with 8 layers, 5.99 with 5 layers at 100K data subset). A full 800K training should exceed this.

### K=128 Evaluation Protocol

1. **Prefix-acceptance curves:** Measure s_i = P(first i tokens accepted) for each K variant
2. **Per-position acceptance rates:** Does alpha remain ~0.87, improve, or degrade at K=128?
3. **End-to-end tau:** Average acceptance length across GSM8K, Math500, HumanEval, MT-Bench
4. **TPU inference speedup:** Use existing standalone benchmark to measure wall-clock improvement

## 10. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| K=16 baseline doesn't reproduce | Low | Blocks everything | Match exact paper setup; use 8-layer model first |
| Per-position alpha degrades with wider blocks | Medium | Negates throughput gain | K=32 and K=64 checkpoints provide intermediate data points |
| γ tuning fails (no good decay schedule) | Low-Medium | Undertrained tail positions | Test uniform weighting; try alternative schedules |
| Anchor overlap issues at K=128 | Low | Training instability | Use non-overlapping (24 anchors) as safe default |
| GPU access unavailable | Medium | Slows development cycle | Fall back to PyTorch/XLA on TPU (Path B) |
| Flex Attention not available | Medium | Need manual mask | Dense mask tensor works; just less memory-efficient |

## 11. Summary

Training DFlash K=128 without the original training code is **feasible and well-scoped**. The architecture is a standard 5-layer transformer (~429M trainable params) with a clear training objective (masked token prediction + position-weighted CE loss). The existing PyTorch model code provides the forward pass; what's missing is the training loop, data pipeline, block construction, and sparse attention mask — all standard ML engineering.

**The main unknowns are not code — they're empirical:**
1. Does per-position acceptance alpha hold, improve, or degrade at K=128?
2. What γ value works for the position-weighted loss at K=128?
3. How many non-overlapping anchors are sufficient per sequence?

The K=16 baseline validation is the critical gate. If your K=16 matches published results, the K=128 experiments are straightforward parameter changes.
