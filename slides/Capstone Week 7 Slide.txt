Problem

The Autoregressive Bottleneck
Standard LLM decoding: 1 forward pass per token
Latency scales linearly with output length
Speculative decoding: cheap draft model proposes tokens, target model verifies in parallel
Key metric: τ = expected accepted tokens per step
Existing drafters (Eagle, Medusa) cap at τ ≈ 2–3

# Phase B: write noise at cache_len + actual_ctx_count,  # overwriting padding zeros from Phase A

kv_cache_k, k_noise_4d, (0, 0, noise_start, 0)

kv_cache_v, v_noise_4d, (0, 0, noise_start, 0)
)
Source: models/jax/dflash.py — context+noise must be written in two phases because context length varies per step.

The seq_len Inflation Bug
Symptom: τ = 2.49, 10% acceptance, 13 A/B tests inconclusive
Root cause: attn_metadata.seq_lens includes unverified draft tokens
Actual generated: 100 tokens
Value passed to proposer: 100 + 16 = 116 (includes draft tokens being verified)
DFlash builds KV cache at wrong position ⇒ stale context accumulates
The 4-line fix
accepted_seq_lens  =  self.runner.input_batch.num_tokens_no_spec[
:attn_metadata.seq_lens.shape[0]].copy()   accepted_attn_metadata = replace(
attn_metadata,  seq_lens=device_array(
self.runner.mesh,   accepted_seq_lens.astype(np.int32)))
Result: τ : 2.49 → 4.48	Speedup: 1.30× → 2.31×

Why the Bug Was Hard to Find
No visible errors — tokens were valid English, just suboptimal; model “recovered” from bad context
Eagle3 unaffected — Eagle3 has no persistent draft-side KV cache, so the same seq_len value worked fine
Gradual accumulation — 10–16 phantom tokens added per decoding step; quality degrades slowly over generations
Bug is upstream — located in the speculative decoding manager, not the DFlash proposer; proposer code was correct

Conclusion

Summary
What We Built
Complete port of DFlash speculative decoding from GPU/PyTorch to TPU/JAX with full vLLM integration and zero upstream changes.
Key Numbers
Standalone: τ = 6.67 (94% of GPU), Math500 exceeds GPU
vLLM pipeline: 2.31× speedup, 212 TPS
DFlash > Eagle3: 2.06× higher τ on TPU
Output quality: 7/8 same answer, 4/8 bit-exact

Key Finding
A 4-line seq_len fix doubled all performance metrics — the hardest bugs leave no trace.
